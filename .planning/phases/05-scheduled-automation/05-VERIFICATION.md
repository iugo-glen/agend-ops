## Gaps Summary

Three truths fail from a single root cause: the live /daily-briefing execution during the scheduled run logged `type: "system"` instead of `type: "briefing"` for the feed entry.

This breaks the downstream pipeline:
1. `scripts/build-dashboard-data.sh` queries `select(.type=="briefing")` — finds nothing — falls back to minimal JSON without a `details` field
2. `docs/briefing.json` ends up as `{"briefing_file": "...", "date": "2026-03-23"}` — no `emails_scanned`, `urgent`, `pending_tasks`, `deadlines_48h`
3. `docs/index.html` banner check `if (briefingResult && briefingResult.details)` evaluates false — banner stays hidden with `--` placeholders

**What's working well:** The briefing file itself (`data/briefings/2026-03-23.md`) is excellent — 100 lines of real, substantive morning briefing with all four required sections. The scheduling infrastructure is proven — a real scheduled run fired and committed. The GitHub Actions fallback is production-ready. The dashboard wiring is correct.

**The fix is minimal:** Correct the existing feed entry type from "system" to "briefing" (single character change in data/feed.jsonl), then re-run `bash scripts/build-dashboard-data.sh`. The dashboard banner will then display correctly on the next page load. No code changes required.
